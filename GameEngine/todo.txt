Custom Game Engine FPS TODO

[현재 상태 요약]
- GameLoop, GameObject, Component의 최소 골격은 있음.
- 실제 구현은 main.cpp 안의 PlayerControl 하나에 입력, 이동, 렌더링이 몰려 있음.
- 렌더링도 2D 삼각형을 매 프레임 버퍼 생성해서 그리는 데모 구조임.
- FPS에 필요한 3D 카메라, Transform, Mesh/Material, Scene 분리가 아직 없음.

[개발 원칙]
- 기능 추가보다 책임 분리를 먼저 한다.
- 2D 과제 구조를 바로 FPS로 확장하지 말고, 3D 엔진의 최소 축을 먼저 세운다.
- "움직이는 3D 테스트맵 + 마우스 룩 + 충돌"까지를 첫 번째 마일스톤으로 잡는다.

[우선순위 TODO]
1. 엔진 기본 구조 재정리
- main.cpp에 몰린 DirectX 초기화, 셰이더 생성, 입력 상태, PlayerControl 정의를 분리한다.
- 최소한 다음 단위로 나눈다:
  Renderer
  Input
  Scene or World
  Camera
  Components
- 목표는 GameLoop가 Input -> Update -> Render 흐름만 관리하도록 만드는 것이다.

2. Transform와 수학 계층 만들기
- Vec3만으로는 FPS 구현이 부족하므로 Transform를 만든다.
- position, rotation, scale을 지원한다.
- 필요 최소 수학 타입:
  Vec2
  Vec3
  Mat4
  Quaternion or Euler rotation
- GameObject가 position만 직접 가지는 구조를 Transform 기반으로 바꾼다.

3. 렌더러를 컴포넌트에서 분리
- 현재 PlayerControl::Render는 매 프레임 VertexBuffer를 생성하므로 구조와 성능 둘 다 좋지 않다.
- Mesh 데이터는 초기화 시 GPU 버퍼를 만들고 재사용하게 바꾼다.
- 다음 클래스를 우선 만든다:
  Mesh
  Material
  MeshRenderer Component
- "입력/이동 로직"과 "그리기"를 완전히 분리한다.

4. 3D 렌더링 최소 기반 구축
- 2D 삼각형 셰이더 대신 월드/뷰/프로젝션 행렬을 쓰는 3D 파이프라인으로 바꾼다.
- 상수 버퍼(Constant Buffer)를 도입한다.
- 정점 포맷도 position/color 수준에서 normal, uv 확장 가능하게 설계한다.
- 첫 단계 목표:
  색상 큐브 1개 렌더링
  바닥 평면 렌더링
  카메라 이동에 따라 화면이 바뀌는 것 확인

5. 카메라 시스템 구현
- FPS의 핵심은 플레이어보다 카메라다.
- FreeCamera부터 만들고, 이후 FPSCamera로 확장한다.
- 필요한 기능:
  yaw/pitch 회전
  view matrix 계산
  projection matrix 계산
  move forward/right/up
- 처음에는 충돌 없는 디버그 카메라로 시작하는 게 맞다.

6. 입력 시스템을 이벤트 캐시에서 액션 기반으로 정리
- 현재 KeyState는 과제용으로는 충분하지만 FPS 확장성은 낮다.
- 키/마우스 입력을 통합해서 관리한다.
- 최소 지원:
  W/A/S/D 이동
  mouse delta로 시점 회전
  ESC 종료
  fullscreen toggle
- 이후 "InputAction" 개념으로 추상화할 수 있게 설계한다.

7. FPS PlayerController 분리
- PlayerControl을 버리고 다음처럼 나눈다:
  PlayerInputComponent
  CharacterController or FPSController
  CameraComponent
- 책임 분리:
  입력 읽기
  이동 계산
  시점 회전
  렌더링은 하지 않음
- 첫 구현에서는 중력/점프 없이 걷기 + 마우스 룩만 먼저 완성한다.

8. Scene/World 관리 추가
- 지금은 GameLoop가 gameWorld 벡터를 직접 들고 있다.
- Scene 클래스를 두고 GameObject 생성, 제거, Update, Render를 맡긴다.
- 이후 맵 로딩, 여러 레벨, 에디터 기능으로 가려면 이 단계가 필요하다.

9. 시간과 물리 기초 정리
- 현재 deltaTime 계산은 들어가 있으므로 방향은 맞다.
- 다음을 보강한다:
  fixed update 여부 결정
  frame rate 표시
  deltaTime clamp
- FPS 조작 안정성을 위해 카메라/플레이어 이동은 큰 dt에서 튀지 않게 해야 한다.

10. 충돌 시스템 최소 구현
- FPS 느낌을 내려면 월드 관통부터 막아야 한다.
- 처음에는 단순한 AABB 충돌로 충분하다.
- 순서:
  static wall collider
  player capsule or AABB
  penetration resolve
- 총알이나 물리보다 플레이어 이동 충돌을 먼저 처리한다.

11. 테스트용 레벨 제작
- 화려한 맵보다 검증용 블록맵이 먼저다.
- 목표 레벨:
  바닥
  벽
  복도
  계단 or 높이 변화는 나중
- 이 레벨에서 이동, 충돌, 카메라, 렌더링을 반복 검증한다.

12. 리소스 로딩 기반 준비
- FPS로 가면 코드 하드코딩 정점만으로는 바로 막힌다.
- 최소한 다음 로딩 경로를 준비한다:
  shader file
  mesh file or primitive factory
  texture
- 첫 단계에서는 큐브/평면 primitive 생성기만 있어도 충분하다.

13. 디버그 도구 추가
- 개발 속도를 크게 좌우한다.
- 최소 기능:
  FPS 표시
  카메라 좌표 출력
  충돌 박스 디버그 드로우
  wireframe toggle
- 엔진 초기에는 기능보다 디버그 가시성이 더 중요하다.

14. 첫 번째 FPS 마일스톤 고정
- 완료 기준:
  3D 큐브/맵 렌더링
  마우스 룩
  WASD 이동
  벽 충돌
  종료/전체화면 동작
- 여기까지 되면 그다음부터 무기, 적, 사운드, 애니메이션을 붙일 수 있다.
